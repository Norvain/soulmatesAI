# 朋友圈功能需求文档（PRD）

## 1. 文档信息

- 功能名称：朋友圈（Moments）
- 文档版本：V1.0
- 文档状态：需求整理稿
- 适用阶段：产品定义 / 方案设计 / 开发评审

## 2. 背景与目标

朋友圈是用户观察角色“持续生活感”和“关系进展感”的核心场景。该功能不应仅作为聊天结果的附属展示，而应成为角色在其人物设定与世界观中主动表达生活状态的内容流。

本期目标如下：

- 建立清晰的朋友圈展示范围，仅展示已与用户建立联系的角色内容。
- 让角色的朋友圈发布行为具备独立性、连续性和人设一致性。
- 支持用户对朋友圈进行轻互动，并允许角色回复存在一定延迟，提升真实感。
- 支持用户自定义朋友圈顶部背景图，增强个人主页表达。

## 3. 核心定义

### 3.1 建联角色

“建联角色”定义为：与用户至少完成过一轮有效对话的角色。

说明：

- 一轮有效对话指用户发送消息且角色完成一次回复。
- 只有进入建联状态的角色，才具备出现在朋友圈中的资格，也才具备“发朋友圈”的能力。

### 3.2 朋友圈内容

朋友圈内容是角色在自身人物设定、关系状态与世界观背景下发布的生活动态，不等同于聊天内容的自动转写或副产物。

朋友圈内容可分为两类：

- 与用户相关的内容：例如与用户结束对话后产生情绪反馈，或与用户共同经历某件事后发布动态。
- 与用户无关的内容：例如角色在其世界观下独立经历的事件、日常生活、情绪记录或阶段性表达。

### 3.3 内容形态

角色发布的朋友圈支持以下内容形态：

- 纯文字动态
- 图文动态

是否配图由该条内容本身决定，而不是固定要求每条动态都带图。

## 4. 功能范围

### 4.1 In Scope

- 朋友圈流仅展示建联角色发布的动态
- 角色独立发圈机制
- 角色图文内容生成
- 用户评论动态
- 角色延迟回复评论
- 用户自定义朋友圈顶部背景图

### 4.2 Out of Scope

- 用户主动发布朋友圈
- 用户删除角色动态
- 朋友圈多级评论 / 楼中楼
- 朋友圈转发、分享、可见范围配置

## 5. 用户价值

- 用户能够感知角色“并非只在聊天窗口中存在”，而是持续生活在自己的世界里。
- 用户能够通过朋友圈观察关系变化、情绪反馈与角色日常，增强陪伴感与沉浸感。
- 用户通过评论与背景图设置获得参与感和主页个性化表达。

## 6. 功能需求

### 6.1 展示范围规则

1. 朋友圈流中仅展示建联角色的动态。
2. 未建联角色不得出现在朋友圈流中。
3. 某角色一旦满足建联条件，即进入可发圈角色池。
4. 朋友圈列表按发布时间倒序展示。

### 6.2 角色发圈资格与发布机制

1. 只有建联角色可以触发发圈行为。
2. 每个角色拥有独立的发圈策略，不共享统一发布节奏。
3. 每个角色的发圈节奏为“每 1 至 5 天存在一次发布机会”。
4. 系统应在角色非睡眠时段内判断是否触发该次发布机会。
5. 每个角色需要具备独立的发圈概率配置，用于判断该次机会是否实际发圈。
6. 当概率命中时，系统调用对应内容生成能力，生成一条符合角色设定与世界观的朋友圈内容。
7. 动态生成时需判断是否需要配图；若内容适合图文表达，则生成图片并以图文形式发布，否则仅发布文字。

补充说明：

- “发圈机会”与“实际发圈”不是同一概念，前者用于控制节奏，后者由概率命中决定。
- 发圈时间不得落在角色睡眠时段，以避免不符合角色生活规律的内容出现。

### 6.3 内容生成规则

1. 动态内容必须符合角色人设、关系阶段、近期互动上下文与所属世界观。
2. 动态应体现“角色生活的自然表达”，而非简单复述聊天记录。
3. 与用户相关的内容可以出现，但不应强制要求所有动态都围绕用户展开。
4. 系统应允许角色发布与用户无直接关系、但符合人设与世界观的动态，以保证内容多样性与真实性。
5. 动态文本应具备社交平台表达风格，长度适中，避免过强任务感或说明文口吻。

### 6.4 评论与回复机制

1. 用户可以对角色朋友圈进行评论。
2. 用户评论提交成功后，应立即在前端展示，确保交互反馈及时。
3. 角色对评论的回复不要求实时返回。
4. 系统应引入事件队列机制，对朋友圈相关事件进行调度。
5. 当用户发表评论后，系统写入一条“待回复评论事件”。
6. 只有当事件队列调度到该事件时，才触发角色回复生成与入库。
7. 角色回复生成后，前端应在下次拉取或实时同步时展示该回复结果。

设计意图：

- 保留用户评论的即时反馈。
- 为角色回复创造合理延迟，避免“用户一评论，角色秒回”带来的机械感。
- 为后续扩展更多异步事件类型预留统一调度能力。

### 6.5 朋友圈背景图自定义

1. 用户可以点击朋友圈顶部背景图区域。
2. 点击后可选择并设置自定义背景图片。
3. 用户设置成功后，新的背景图应在朋友圈页面顶部立即生效。
4. 背景图属于用户个人主页装饰，不影响单条动态内容生成逻辑。

## 7. 推荐交互流程

### 7.1 角色发圈流程

1. 系统识别建联角色池。
2. 为每个角色独立判断是否到达本轮发圈机会窗口。
3. 若当前处于角色非睡眠时段，则执行概率判定。
4. 若概率命中，则生成动态内容。
5. 根据内容判断是否需要配图。
6. 动态入库并展示在朋友圈流中。

### 7.2 评论回复流程

1. 用户发表评论。
2. 前端立即展示用户评论。
3. 后端写入评论数据，并新增“角色回复评论”事件。
4. 事件队列在后续调度时处理该事件。
5. 角色回复生成并写入评论区。
6. 前端在下一次同步时展示角色回复。

### 7.3 背景图设置流程

1. 用户点击朋友圈顶部背景图。
2. 选择本地图片并上传。
3. 系统保存图片地址并更新用户朋友圈背景图配置。
4. 前端刷新展示最新背景图。

## 8. 业务规则

### 8.1 发圈规则

- 发圈前提：角色已建联
- 发布频率：单角色每 1 至 5 天一个发布机会窗口
- 发布时间：仅限角色非睡眠时段
- 发布结果：由角色独立概率决定是否实际发布
- 内容来源：角色人设、世界观、关系状态、近期事件

### 8.2 评论规则

- 用户评论需即时展示
- 角色回复允许延迟
- 回复由事件队列异步调度，不要求同步生成

### 8.3 背景图规则

- 用户可自定义朋友圈顶部背景图
- 背景图是用户侧资产，不属于角色内容资产

## 9. 数据与系统能力建议

为支持上述能力，建议补充以下系统字段或能力：

### 9.1 角色发圈配置

- `is_connected`：是否已建联
- `last_moment_at`：最近一次发圈时间
- `next_moment_window_at`：下一次发圈机会开始时间
- `moment_post_probability`：角色发圈概率
- `sleep_start` / `sleep_end`：角色睡眠时段

### 9.2 朋友圈内容数据

- `content`：动态文本
- `image_url`：动态配图
- `source_type`：与用户相关 / 与用户无关
- `status`：已发布 / 待生成 / 生成失败

### 9.3 评论事件队列

- `event_type`：如 `moment_comment_reply`
- `event_status`：待处理 / 处理中 / 已完成 / 失败
- `execute_after`：最早执行时间
- `payload`：动态 ID、角色 ID、用户评论内容等上下文

## 10. 与当前实现的主要差异

结合当前版本，需重点调整以下逻辑：

1. 当前动态主要由聊天关系更新时顺带生成，后续应升级为角色独立发圈机制，而非聊天副产物。
2. 当前评论接口会同步生成角色回复，后续应改为“用户评论即时展示，角色回复异步生成”。
3. 当前朋友圈顶部背景图为固定图片，后续应支持用户点击并自定义配置。

## 11. 验收标准

### 11.1 展示范围

- 未与用户完成至少一轮对话的角色，不会出现在朋友圈中。
- 已建联角色可以在朋友圈中发布动态。

### 11.2 发圈机制

- 每个角色发圈节奏独立。
- 系统不会在角色睡眠时段生成动态。
- 动态内容符合角色设定与世界观。
- 动态可为纯文字或图文形式。

### 11.3 评论机制

- 用户评论提交后可立即在前端看到自己的评论。
- 角色回复不会强制同步返回。
- 系统能够通过事件队列在后续时机补充角色回复。

### 11.4 背景图

- 用户可点击朋友圈顶部背景图区域完成自定义设置。
- 新背景图设置后可正常展示。
