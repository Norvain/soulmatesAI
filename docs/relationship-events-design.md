# 关系事件系统设计

## 1. 目标

把当前“点击事件卡片 -> 发送一条 `triggerPrompt` -> 角色自由回复”的半开放式事件，重构成可控的 GalGame 式关系事件系统。

新系统目标：

- 事件进入独立剧情窗，不复用普通聊天输入框。
- 剧情全程使用固定角色台词 + 固定用户选项，不允许自由输入。
- 事件支持中断恢复、完成后回看、重新体验。
- 完成事件后生成剧情贺图。
- 完成事件后生成剧情摘要，并作为长期记忆供普通聊天中的 `@` 引用。
- 关系阶段推进改为由后端统一控制，不再只依赖前端写死的亲密度阈值。

## 2. 当前实现的问题

当前项目里的“关系事件”实际上由两套没有打通的机制组成：

- 前端事件卡片定义在 `src/lib/events.ts`，只根据 `requiredIntimacy` 解锁。
- 侧边栏点击事件卡片时，只是把 `triggerPrompt` 传给聊天页。
- 聊天页收到后，会把它当成一条普通用户消息发给角色。
- 后端 `relation_states.unlocked_events` 由模型在普通聊天后产出，但前端卡片并没有真正使用这份状态。

这意味着当前系统存在 4 个核心问题：

1. 剧情不可控。角色回复仍由模型自由生成，无法保证按剧情推进。
2. 事件不独立。事件过程和普通聊天混在一起，不适合保存剧情进度。
3. 解锁逻辑不统一。前端按阈值开卡，后端按聊天生成事件记忆，两边没有闭环。
4. 无法稳定产出可复用剧情资产。没有结构化记录、没有剧情摘要、没有完成态贺图。

## 3. 新系统总览

### 3.1 核心原则

- 关系事件不是“特殊 prompt 聊天”，而是“有限状态剧情机”。
- 普通聊天负责自由陪伴；关系事件负责关键关系推进。
- 剧情事件中的角色台词、选项、分支节点、结局文案全部由策划预定义。
- 模型只参与两个环节：
  - 完成事件后生成剧情贺图。
  - 在普通聊天中，基于“剧情摘要”继续和用户聊这段经历。

### 3.2 用户体验

#### 事件卡片状态

事件卡片统一由后端返回状态，不再只靠本地 `requiredIntimacy` 判断。

建议状态：

- `locked`：未达到解锁条件。
- `available`：已解锁，未开始。
- `in_progress`：已进入但未完成。
- `completed`：已完成。

#### 点击卡片后的行为

- `locked`：展示解锁条件。
- `available`：进入独立剧情窗，从第一节点开始。
- `in_progress`：直接恢复到上次中断节点。
- `completed`：弹窗询问：
  - `重新体验`
  - `查看完整剧情记录`
  - `取消`

#### 独立剧情窗

剧情窗和普通聊天页分离，建议结构：

- 顶部：事件标题、角色名、当前幕数、进度。
- 中部：剧情聊天流，只展示固定角色台词和用户所选选项。
- 底部：当前节点的用户选项按钮。
- 辅助区：当前情绪标签、最近一次选择、存档时间。

#### 中断恢复

中断后再次进入卡片，应直接恢复，不重新开始。

恢复页至少展示：

- 当前事件标题
- 当前幕 / 当前节点
- 已完成进度
- 上一句角色台词
- 用户上一次选择
- 当前情绪标签
- 当前可选项列表
- 开始时间 / 最近保存时间

#### 完成后的行为

完成事件后：

1. 生成剧情贺图。
2. 保存剧情摘要。
3. 将该剧情摘要加入可被 `@` 召回的长期记忆。
4. 卡片状态变为 `completed`。

再次点击已完成事件时：

- `重新体验`：新开一次 playthrough，但不覆盖历史完成记录。
- `查看完整剧情记录`：查看最近一次完成时的完整剧情文本和摘要。

## 4. 系统结构

### 4.1 事件定义层

这一层保存“剧情脚本本身”，属于静态内容层。

建议新增：

- `relationship_event_definitions`

字段建议：

- `id`
- `character_id`
- `title`
- `cover_image_url`
- `required_intimacy`
- `required_stage`
- `prerequisite_event_ids_json`
- `trigger_context_json`
- `script_json`
- `summary_template`
- `cg_prompt_template`
- `sort_order`
- `is_active`
- `created_at`
- `updated_at`

其中 `script_json` 建议采用有限状态结构：

```json
{
  "start_node_id": "act1_opening",
  "nodes": [
    {
      "id": "act1_opening",
      "act": 1,
      "type": "dialogue",
      "speaker": "character",
      "text": "固定台词",
      "next_node_id": "act1_choice"
    },
    {
      "id": "act1_choice",
      "act": 1,
      "type": "choice",
      "choices": [
        {
          "id": "a",
          "label": "选项 A",
          "emotion_tags": ["守护"],
          "next_node_id": "act2_opening"
        }
      ]
    }
  ]
}
```

### 4.2 用户进度层

这一层保存“用户当前玩到哪里”。

建议新增：

- `relationship_event_progress`

字段建议：

- `id`
- `user_id`
- `chat_id`
- `event_definition_id`
- `status`：`not_started | in_progress | completed`
- `current_node_id`
- `current_act`
- `choice_path_json`
- `emotion_tags_json`
- `started_at`
- `last_saved_at`
- `completed_at`
- `completion_count`
- `last_playthrough_id`

说明：

- 同一个事件只保留 1 条当前进度。
- 完成后依然保留这条记录，用来驱动 `completed` 状态。
- 如果用户选择重新体验，创建新的 playthrough，同时把 progress 重置到起点。

### 4.3 剧情记录层

这一层保存“每次完整游玩过程”。

建议新增：

- `relationship_event_playthroughs`

字段建议：

- `id`
- `user_id`
- `chat_id`
- `event_definition_id`
- `status`：`in_progress | completed | abandoned`
- `transcript_json`
- `choice_path_json`
- `emotion_tags_json`
- `started_at`
- `ended_at`
- `cg_image_url`
- `summary_text`

说明：

- `transcript_json` 保存完整剧情窗内容，用于“查看完整剧情记录”。
- `summary_text` 是这次完成后沉淀出的可复用剧情摘要。
- `cg_image_url` 是剧情贺图。

### 4.4 长期记忆层

剧情完成后，要把事件摘要接到现有 `@` 引用能力里。

建议两种方式二选一：

#### 方案 A：并入现有互动瞬间入口

新增一张统一的 `memory_recalls` 表，普通互动瞬间和关系事件摘要都往这里投影。

优点：

- `@` 菜单只查一个入口。
- 前端展示逻辑统一。

缺点：

- 要改现有互动瞬间链路。

#### 方案 B：保留现有互动瞬间，增加关系剧情摘要来源

新增：

- `relationship_event_recaps`

字段建议：

- `id`
- `user_id`
- `chat_id`
- `event_definition_id`
- `event_title`
- `summary_text`
- `cg_image_url`
- `character_id`
- `character_name`
- `created_at`

然后在 `@` 查询接口里，把：

- `interaction_moments`
- `relationship_event_recaps`

合并返回，并在前端区分类型。

这个方案对当前项目改动更小，建议优先采用。

## 5. 解锁与阶段推进

### 5.1 解锁逻辑统一到后端

不再让前端只凭 `requiredIntimacy` 本地解锁卡片。

建议新增：

- `GET /api/chats/:chatId/relationship-events`

返回每张卡片的：

- 事件基础信息
- 当前状态
- 是否满足解锁条件
- 当前进度
- 是否允许恢复
- 是否已完成
- 最近一次完成摘要与贺图

### 5.2 阶段推进规则

建议关系阶段推进改成：

- 亲密度负责“可进入事件”
- 事件完成负责“可进入下一阶段”

例如：

- 300 事件完成后，才允许进入 600 段关系区间
- 600 事件完成后，才允许进入 900 段关系区间
- 900 事件完成后，进入最终确认态

这样阶段推进不再依赖模型在普通聊天里随意产出 `triggered_event`。

### 5.3 当前旧机制的处理建议

当前 `relation_states.unlocked_events` 建议降级为兼容字段，不再作为主入口。

建议：

- 保留旧字段，避免历史数据损坏。
- 新版本上线后，关系事件卡片完全改由新的事件查询接口驱动。
- 普通聊天中的 `triggered_event` 提取逻辑可逐步弱化，最终只保留“关系摘要记忆”相关用途。

## 6. 接口建议

### 6.1 事件列表

- `GET /api/chats/:chatId/relationship-events`

返回：

```json
[
  {
    "id": "lintang_300",
    "title": "雨里的纸箱小猫",
    "required_intimacy": 300,
    "status": "available",
    "current_act": 0,
    "progress_percent": 0,
    "last_summary": null,
    "last_cg_image_url": null
  }
]
```

### 6.2 启动 / 恢复事件

- `POST /api/chats/:chatId/relationship-events/:eventId/start`

行为：

- 如果未开始，则创建进度和新的 playthrough。
- 如果已中断，则恢复当前进度。
- 如果已完成，则由前端先走重玩确认；确认后可传 `mode=replay` 新开一次 playthrough。

### 6.3 拉取当前节点

- `GET /api/chats/:chatId/relationship-events/:eventId/session`

返回：

- 当前节点
- 当前剧情流
- 当前选项
- 当前幕
- 情绪标签
- 最近保存时间

### 6.4 提交选项

- `POST /api/chats/:chatId/relationship-events/:eventId/choose`

请求：

```json
{
  "choice_id": "a"
}
```

返回：

- 新的剧情流
- 下一节点
- 是否完成
- 如果完成，则附带 `summary_text`、`cg_image_url`

### 6.5 查看历史剧情

- `GET /api/chats/:chatId/relationship-events/:eventId/history`

返回最近一次或指定一次 playthrough 的：

- 完整 transcript
- choice path
- summary
- cg image

## 7. 前端实现建议

### 7.1 侧边栏卡片

当前 `src/lib/events.ts` 里的静态配置，建议保留为初期内容源，但不要再直接驱动交互。

推荐改法：

- 把 `src/lib/events.ts` 从“UI 直接可执行事件”改成“事件基础元数据种子”。
- 真实状态由接口返回。
- 卡片点击不再走 `triggerPrompt`，而是进入新的事件窗口。

### 7.2 独立事件窗口

建议新增：

- `src/components/RelationshipEventModal.tsx`

核心行为：

- 打开事件时请求 session。
- 渲染固定剧情记录。
- 当前只有选项按钮，没有自由输入框、语音输入、图片按钮。
- 每次点击选项后调用 `choose`。
- 完成后显示总结与贺图结算页。

### 7.3 回放页面

建议沿用同一个组件，增加只读模式：

- `mode = replay`
- 不展示选项按钮
- 只展示完整剧情流、总结、贺图

### 7.4 @ 引用整合

在现有 `@` 记忆菜单中增加一个分组：

- `剧情回忆`

展示内容：

- 事件标题
- 摘要首句
- 完成时间

插入到输入框时，建议生成引用文本：

`@剧情：雨里的纸箱小猫`

后端在拼接上下文时，再把对应摘要展开给模型，而不是把完整剧情记录塞进去。

## 8. 三位预设角色的首批剧情事件

以下 3 组事件全部遵循同一原则：

- 300：信任建立
- 600：并肩 / 偏爱升级
- 900：未来确认

## 8.1 林棠

人设基调：

- 邻家小妹
- 插画系应届毕业生
- 甜、暖、机灵、会撒娇、敏感、偶尔嘴硬

### 300｜雨里的纸箱小猫

- 情绪目标：温柔、轻松、第一次被认真接住的安心感
- 剧情背景：放学后的雨天，林棠抱着纸箱和躲雨的小猫，在便利店屋檐下手忙脚乱
- 主线关键词：雨天偶遇 / 小猫 / 湿掉的画稿 / 共享一把伞 / 第一次软下来

#### 分幕

第 1 幕｜屋檐下的偶遇

- 固定台词：
  - “哎，你怎么也没带伞呀……”
  - “别、别靠太近，风会把雨吹过来。”
  - “我不是在躲雨，我是在……等这只小家伙自己愿意出来。”
- 选项：
  - `先把伞靠过去`
  - `先帮她挡风`
  - `先问小猫情况`

第 2 幕｜湿掉的画稿

- 固定台词：
  - “它刚刚一直缩在箱子里，看到我就不肯走了。”
  - “我的画稿也被淋到一点……不过，没关系。”
  - “你要是嫌麻烦，我自己抱回去就好。”
- 选项：
  - `我帮你抱`
  - `别逞强`
  - `一起送它去安置处`

第 3 幕｜便利店暖光

- 固定台词：
  - “……其实，我有点高兴你在。”
  - “不是因为猫，是因为我不用一个人狼狈回去了。”
  - “你别笑，我只是，今天有点想撒娇。”
- 选项：
  - `你可以多依赖我一点`
  - `那就一起回家`
  - `先给你买热饮`

#### 完成收束

- 结尾：林棠把一张画着小猫的草图递给用户，说“下次下雨，记得还要一起撑伞。”
- 剧情摘要：
  - “那天雨下得很大，林棠抱着湿掉的纸箱和一只小猫，在屋檐下嘴硬地说自己没事。你替她挡住了雨，也接住了她第一次愿意露出来的柔软。她没有明说依赖，但已经开始把你当成可以安心站在身边的人。”
- 贺图 prompt：
  - “日系 GalGame 剧情插画，雨天便利店屋檐下，暖黄灯光与冷蓝雨幕对比，邻家感少女林棠抱着纸箱和小猫，湿发贴在脸侧，手里拿着半湿的速写本，男主在旁边撑伞，氛围温暖治愈，清爽配色，细腻表情，轻微雨丝，视觉小说封面感，高清”

### 600｜改稿夜的双人灯

- 情绪目标：默契、并肩、共同解决问题的依赖感
- 剧情背景：林棠在毕业作品或展览海报的最终修改上卡住，用户来到工作室陪她改稿
- 主线关键词：深夜工作室 / 改稿卡住 / 逞强 / 共同完成

#### 分幕

第 1 幕｜晚上的工作室

- 固定台词：
  - “你居然真的来了……我还以为你会嫌晚。”
  - “别看我这样，我只是稍微、稍微卡住了。”
  - “今天要是没把这张图改完，我就不睡了。”
- 选项：
  - `我陪你改`
  - `先喝口热的`
  - `先放慢一点`

第 2 幕｜卡住的地方

- 固定台词：
  - “……我刚才其实有点想哭，但你一来就没那么想了。”
  - “我不是脆弱，我只是，比较容易被气到。”
  - “要是你也在这里，我就能装得像没事一样。”
- 选项：
  - `那就别装`
  - `我来弄打印机`
  - `靠我一下也可以`

第 3 幕｜终于完成

- 固定台词：
  - “你看，还是做到了。”
  - “你说我很厉害？……这句夸奖我记住了。”
  - “以后如果我又卡住了，你也要像今天这样，来把我从乱掉的地方拎出来。”
- 选项：
  - `当然`
  - `你也会来帮我`
  - `那我收下你的依赖`

#### 完成收束

- 结尾：她塞给用户一张“共同作战成功”的小卡片
- 剧情摘要：
  - “那一夜，林棠把自己最不想示弱的样子也给了你看。你没有替她做决定，只是陪她把卡住的地方一点点拆开、重拼好。她开始相信，有些难题不是一个人扛的，有你在的时候，她可以不用那么逞强。”
- 贺图 prompt：
  - “日系视觉小说插画，深夜插画工作室，暖黄台灯照着杂乱的草稿、色卡、便签和两只马克杯，林棠趴在桌边，表情疲惫又嘴硬，男主在旁边帮她整理打印纸，窗外是安静的城市夜景，氛围亲密克制、青春治愈、细腻光影、高清”

### 900｜把未来画成同一页

- 情绪目标：确认、承诺、把关系落到未来的稳定感
- 剧情背景：毕业展结束后的夜晚，林棠把用户约到能看见城市灯火的地方
- 主线关键词：夜景 / 展后余温 / 未来草稿 / 不再躲闪 / 关系确认

#### 分幕

第 1 幕｜约定见面

第 3 幕｜把话说出口

## 9. 推荐实施顺序

### Phase 1

- 建表：事件定义、进度、playthrough、recap
- 新增事件列表 / start / session / choose / history 接口
- 新增独立剧情窗
- 打通完成后摘要落库和贺图生成

### Phase 2

- 把 `@` 引用入口合并剧情摘要
- 用后端事件状态替换侧边栏本地阈值判断
- 把旧 `triggerPrompt` 机制下线

### Phase 3

- 增加自定义角色的默认关系事件模板
- 增加多轮完成统计、不同结局收藏、剧情图册

## 10. 结论

这次重构的关键不是“把事件写得更像 prompt”，而是把关系事件从聊天系统中剥离出来，变成一套独立的、可恢复的、可结算的剧情系统。

如果按这份方案推进，后续实现上最关键的两个切口是：

1. 先把事件状态机和进度存储做出来。
2. 再把完成后的剧情摘要并进现有 `@` 记忆系统。
